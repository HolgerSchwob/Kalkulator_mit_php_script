Abstract & Projektvision

Dieses Dokument beschreibt die Architektur und Funktionalität des Kalkulator 2.0, einer hochmodernen Web-Anwendung zur Konfiguration und Personalisierung von Druckprodukten. Die Kernvision des Projekts war die Schaffung einer nahtlosen User Experience, die einen leistungsstarken Online-Kalkulator mit einem intelligenten SVG-Buchdecken-Editor vereint.

Der Kunden-Workflow

Der Prozess für den Kunden ist linear und intuitiv gestaltet, um ihn von der ersten Idee bis zur fertigen Anfrage zu führen.

  1. 1. Buchblock konfigurieren: Der Kunde gibt Seitenzahl, Papiersorte und Druckmodus an.
  2. 2. Bindungsvarianten wählen: Der Kunde fügt eine oder mehrere Bindungsvarianten hinzu.
  3. 3. Personalisieren: Für Bindungen, die dies erfordern, startet der Kunde den Editor, um die Optik zu gestalten.
  4. 4. Extras & Lieferung: Weitere Optionen werden gewählt.
  5. 5. Angebot & Anfrage: Der Kunde kann ein PDF-Angebot herunterladen oder den Anfrageprozess starten.

Technische Architektur

Das Herzstück der Anwendung ist die Architektur des "Dirigenten und der Spezialisten".

Das Prinzip

  • Der Dirigent (script.js): Die zentrale Steuereinheit. Sie besitzt den alleinigen, globalen Zustand (inquiryState) und ist die "Single Source of Truth".
  • Die Spezialisten (*.mjs): Jedes Modul hat eine klar definierte Aufgabe. Es verwaltet keinen eigenen Zustand.

Die config.json - Das Herzstück

Alle geschäftsrelevanten Daten sind in der config.json zentralisiert.

Best Practice: JSON Schema

Für zukünftige Entwicklungen wird die Verwendung eines JSON Schemas empfohlen. Ein Schema definiert formal die erwartete Struktur und die Datentypen der config.json. Dies verhindert Fehler durch Tippfehler (z.B. ein String statt einer Zahl bei einem Preis) und dient als "lebende Dokumentation" der Konfigurationsmöglichkeiten.

Preisfindungs-Algorithmus

Die Preisberechnung verfolgt die "Smarte & ehrliche Methode": maximale Transparenz für den Kunden bei gleichzeitiger Abbildung der internen Kostenstruktur.

Die Berechnung des Stückpreises im Warenkorb

Um dem Kunden einen einfachen und verständlichen Preis zu präsentieren, werden die Kosten für Buchblock und Bindung zu einem einzigen Stückpreis pro Variante zusammengefasst. Dieser Stückpreis berücksichtigt die Menge, um einen fairen Rabatt-Effekt zu erzielen:

Stückpreis = (Variable_Kosten_Buchblock + Variable_Kosten_Bindung) + ((Grundpauschale_Buchblock + Grundpauschale_Bindung) / MENGE)

SVG-Editor Deep Dive

Die Intelligenz des Editors liegt darin, dass die Design-Logik vollständig in den SVG-Templates selbst definiert ist.

Nomenklatur: Die Sprache der SVGs

Damit der Editor ein Template versteht, muss eine strikte Namenskonvention für die Element-IDs eingehalten werden:

  • tpl-group-*: Gruppen für U1, U4 und Buchrücken.
  • tpl-*: Präfix für steuerbare Elemente (Texte, Logos).
  • *_bbox: Ein unsichtbares Rechteck als Begrenzungsrahmen für Text.

Behandlung von Grenzfällen (Edge Cases)

  • Zu langer Text: Wird für einen Text eine _bbox definiert, versucht der Editor, den Text durch Skalierung in die Box einzupassen. Unterschreitet die nötige Skalierung ein Minimum (z.B. 75%), wird der Text mit der Minimalskalierung dargestellt und eine sichtbare Warnung im SVG eingeblendet.
  • Zeilenumbrüche: Textfelder, die im SVG das Attribut data-multiline="true" tragen, werden im Editor als <textarea> dargestellt. Der Editor erzeugt für jeden Zeilenumbruch ein neues <tspan>-Element und stellt so die korrekte Darstellung im SVG sicher.

Wartung & Erweiterung

Dank der modularen Architektur sind Anpassungen und Erweiterungen einfach und risikoarm durchzuführen.

Der "Config-First"-Ansatz: Ein Editor für alles

Die Weiterentwicklung des Editors folgt einem klaren Prinzip: Es wird nur einen einzigen CoverEditor.mjs geben. Anstatt für jeden Bindungstyp eine neue Klasse zu erstellen, wird das Verhalten des Editors ausschließlich über Konfigurationsprofile und die SVG-Templates selbst gesteuert. Dies ermöglicht maximale Flexibilität und Wartbarkeit.

Editor-Profile in config.json

Jeder Bindungstyp, der den Editor verwendet, verweist auf ein Profil im neuen editorProfiles-Objekt der config.json. Dieses Profil ist der Bauplan für den Editor:

"hardcover-modern": {
    "templatePath": "templates/hardcover/",
    "dimensions": { "width": 215, "height": 302 },
    "features": ["textInput", "logoUpload", "spineText"]
},
"paperback-foil": {
    "templatePath": "templates/paperback-foil/",
    "dimensions": { "width": 210, "height": 297 },
    "features": ["acceptsPdfThumbnail", "hasFoilOverlay"]
},
"wire-o": {
    "templatePath": "templates/wire-o/",
    "dimensions": { "width": 210, "height": 297 },
    "features": ["acceptsPdfThumbnail", "dynamicSpineVisuals"],
    "spineVisualRules": [
      { "maxThickness": 12, "elementId": "wire-o-3-to-1" },
      { "maxThickness": 999, "elementId": "wire-o-2-to-1" }
    ]
}

Implementierungs-Beispiele

  • Paperback mit Folie: Das Profil aktiviert das Feature acceptsPdfThumbnail. Der editorHandler übergibt dem Editor die URL des PDF-Thumbnails. Der Editor zeigt dann eine UI-Option an, um dieses Thumbnail auf einem vordefinierten Platzhalter im SVG (z.B. <rect id="u1-content-placeholder" />) zu platzieren. Das Feature hasFoilOverlay wird durch einen Radio-Button im Haupt-Kalkulator gesteuert und schaltet die Sichtbarkeit eines entsprechenden <rect id="foil-overlay" /> im Template um.
  • Wire-O: Das Profil aktiviert dynamicSpineVisuals und liefert ein Regelset. Der Editor liest die übergebene Rückenbreite, vergleicht sie mit den maxThickness-Werten und blendet die korrekte SVG-Gruppe für den Drahtkamm (z.B. <g id="wire-o-3-to-1">...</g>) ein, während andere ausgeblendet werden.
